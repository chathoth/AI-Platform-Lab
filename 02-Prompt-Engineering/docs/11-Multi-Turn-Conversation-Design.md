# 11 - Multi-Turn Conversation Design

## Introduction

Module 01 chapter 08 established that a model has no memory between
calls — the application resends history every turn. This chapter is
about *designing* that resend deliberately, instead of just appending
every message forever and hoping it stays coherent and affordable.

## Learning Objectives

After this chapter I should be able to:

-   Design a conversation history structure that stays within budget as
    it grows.
-   Decide what to keep, summarize, or drop from history.
-   Recognize when a "conversation" should actually be a fresh call
    instead.

------------------------------------------------------------------------

# The Problem: History Grows, Forever, By Default

``` python
history = [system_prompt]
# turn 1: history = [system, user1, assistant1]           - 3 messages
# turn 10: history = [system, user1, assistant1, ..., user10, assistant10]  - 21 messages
```

Every turn re-sends the *entire* history (module 01 chapter 09) — so
turn 10 of a conversation costs meaningfully more tokens than turn 1,
even if the actual question is just as short. Left unmanaged, a long-
running chat session slowly turns into the most expensive, slowest call
in the system, and eventually risks exceeding the context window
outright.

**Platform analogy:** unmanaged conversation history is an unbounded
log buffer — fine at first, then it quietly becomes the thing that
blows your memory limit at 2am. The fix is the same: a retention
policy, not "keep everything forever and hope."

## Retention Strategies

  Strategy               How it works                                   Trade-off
  ----------------------- ----------------------------------------------- --------------------------------
  Sliding window           Keep only the last N turns                     Loses early context entirely
  Summarize-and-compact     Periodically replace old turns with a summary  Extra LLM call, some detail lost
  Pin + trim                 Keep system prompt + first turn + last N turns Balances "why we started" with recency
  Fresh call per task        Don't treat it as a conversation at all         No history cost, but no continuity

``` python
def trim_history(history, system_prompt, keep_last=6):
    # Always keep the system prompt, drop everything except the most
    # recent `keep_last` messages - a sliding window.
    recent = history[-keep_last:]
    return [system_prompt] + recent
```

``` python
def summarize_and_compact(history, client, model):
    # Replace everything except the last 2 messages with a short
    # summary, generated by the model itself.
    old, recent = history[:-2], history[-2:]
    summary_prompt = "Summarize this conversation in 2-3 sentences, keeping only decisions and facts that matter for what follows:\n" + \
        "\n".join(f"{m['role']}: {m['content']}" for m in old)
    summary = client.chat.completions.create(
        model=model, messages=[{"role": "user", "content": summary_prompt}], temperature=0
    ).choices[0].message.content
    return [{"role": "system", "content": f"Earlier conversation summary: {summary}"}] + recent
```

## Not Everything Needs to Be a Conversation

If each user request is actually independent — classify this alert,
summarize that log — forcing it through a shared conversation history
adds cost and risk (stale context leaking into an unrelated answer) for
no benefit. Reserve multi-turn design for genuinely stateful
interactions (a chatbot, an iterative debugging session) and use a
single fresh call for anything that doesn't need what came before.

## Hands-on: Watch the Cost Curve

``` python
import tiktoken
from openai import OpenAI

client = OpenAI(base_url="http://localhost:11434/v1", api_key="ollama")
MODEL = "llama3.1:8b"
enc = tiktoken.encoding_for_model("gpt-4")

history = [{"role": "system", "content": "You are a concise DevOps assistant."}]
questions = [
    "What is a readiness probe?",
    "How is it different from a liveness probe?",
    "What happens if I don't set one?",
    "Give an example YAML snippet.",
]

for q in questions:
    history.append({"role": "user", "content": q})
    total_tokens = sum(len(enc.encode(m["content"])) for m in history)
    print(f"Before call: {len(history)} messages, ~{total_tokens} tokens going in")
    r = client.chat.completions.create(model=MODEL, messages=history, temperature=0.3)
    history.append({"role": "assistant", "content": r.choices[0].message.content})
```

Watch the token count climb turn over turn, purely from re-sending
history — then try inserting `trim_history()` after turn 2 and see the
count reset.

## Common Misconceptions

❌ Keeping full history is "safer" because nothing gets lost.
(Unbounded history is a cost and context-window risk, and per module 01
chapter 09's "lost in the middle" problem, older context in a huge
history is used *less* reliably anyway — not a pure win.)

❌ Every interaction with an LLM should be a conversation.
(Independent, stateless requests are simpler, cheaper, and safer as
individual calls — multi-turn design is for genuinely stateful tasks.)

✔ Conversation history needs an explicit retention policy — sliding
window, summarization, or pin-and-trim — the same way any growing log
or buffer needs one.

## Interview Questions

3.  When should an interaction be a fresh call instead of part of a
    conversation?
4.  What's the trade-off of a sliding-window retention strategy?

## Summary

Conversation history grows without bound by default because every turn
resends everything before it — the fix is an explicit retention policy
(sliding window, summarization, or pin-and-trim), chosen deliberately
rather than left to accumulate. Not every interaction needs to be
stateful in the first place; independent requests are cheaper and
safer as fresh calls.
